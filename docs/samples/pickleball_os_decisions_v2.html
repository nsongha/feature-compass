1. Referee App: migrate sang Supabase hay giữ local + sync layer?
Critical▼
Rewrite với Supabase integration
Referee App nhận match_id từ Admin Web → ghi scores vào Supabase → Admin Web nhận qua Supabase Realtime. Sạch nhất về architecture.
Recommended — đã chốt Supabase Realtime
Giữ local + thêm sync layer
Giữ toàn bộ scoring logic hiện tại. Thêm một sync module: khi online → POST score lên Supabase. Offline-first tốt hơn.
Phức tạp hơn về conflict resolution
Adapter pattern — wrap, không rewrite
Scoring engine giữ nguyên. Thêm một adapter nhận match context từ Admin Web và push kết quả lên Supabase sau trận.
Fastest — nhưng không thực sự realtime
Kết nối với: Referee App hiện có match log + rally stats rất chi tiết (ace, smash, kitchen fault...) — cần quyết định level detail nào được sync lên Supabase (câu hỏi 9).
Ảnh hưởng: API_CONTRACT.md endpoint design, DATA_SCHEMA matches table, offline sync strategy, conflict resolution khi 2 trọng tài cùng nhập.
Ghi chú
2. Athlete App: Local Storage → Supabase — migrate toàn bộ hay hybrid?
Critical▼
Full migration → Supabase
Tất cả data (match log, skills tree, gear, health) lên Supabase. Player đăng nhập ở đâu cũng thấy data. Admin Web đọc được skills để suggest category phù hợp.
Recommended — unlock cross-app features
Hybrid — match history cloud, personal data local
Match results sync lên Supabase. Skills tree, gear, health vẫn Local Storage — đây là data cá nhân, không cần Admin Web đọc.
Đơn giản hơn, nhưng DUPR không tính được từ tournament data
Kết nối với: Athlete App tính DUPR ước tính từ skills tree — nếu migrate toàn bộ, Admin Web có thể feed tournament match results vào DUPR calculation tự động, chính xác hơn nhiều so với self-report hiện tại.
Ảnh hưởng: players table schema, auth (player tự đăng ký hay Admin Web tạo trước), DUPR calculation location.
Ghi chú
3. Shared auth: 1 Supabase account cho cả 3 apps hay auth riêng?
Critical▼
Shared Supabase auth — role-based JWT
Cùng 1 user_id xuyên suốt. JWT custom claim: role: admin | referee | athlete. RLS policy dùng role để phân quyền data.
Recommended — đơn giản nhất cho solo builder
Auth riêng từng app
Mỗi app manage session riêng. Referee App dùng PIN/code ngắn do Admin Web cấp — không cần tài khoản đầy đủ.
UX phức tạp hơn, data join khó
Shared auth + Referee dùng match code
Admin Web & Athlete App dùng Supabase Auth đầy đủ. Referee App: Admin Web sinh code 6 ký tự → trọng tài nhập vào → truy cập match không cần account.
UX tốt nhất cho referee
Ảnh hưởng: AUTH_FLOW.md toàn bộ, RLS policies, user onboarding cho từng role, session management.
Ghi chú
4. Player profile: ai tạo trước — Admin Web hay Athlete tự đăng ký?
Important▼
Admin Web tạo trước → Athlete claim sau
BTC nhập danh sách VĐV khi tạo giải. Mỗi VĐV nhận email/SMS với link claim profile. Athlete App link vào profile đó.
Recommended — Admin kiểm soát tournament roster
Athlete tự đăng ký → BTC approve
VĐV tự tạo profile trên Athlete App. BTC tìm kiếm và invite vào giải. Linh hoạt hơn nhưng BTC khó quản lý roster.
Mất kiểm soát eligibility
Kết nối với: Athlete App hiện có player name nhưng không có player_id. Referee App cũng chỉ có tên — không có ID. Sau khi có shared player_id, Referee App tự động biết tên VĐV từ Admin Web, không cần nhập lại.
Ảnh hưởng: players table, registration flow, invite system, how Referee App gets player names.
Ghi chú
5. DUPR / ranking: tính ở đâu?
Important▼
Supabase Edge Function — server-side
Trigger sau khi Referee App submit score → Edge Function tính DUPR delta → cập nhật players.rating. Accurate hơn nhiều so với self-report skills.
Recommended — dùng real match data
Giữ trong Athlete App — self-assessment
Skills tree 0–10 như hiện tại. Tournament match results được import vào Athlete App để làm đầu vào tính DUPR estimate, nhưng logic vẫn client-side.
Kết nối với: Athlete App hiện tính DUPR ước tính từ skills tree completion rate. Nếu dùng Edge Function với real match data (win/loss vs. rated opponents), DUPR estimate sẽ sát với DUPR chính thức hơn nhiều — đây là differentiator lớn.
Ảnh hưởng: players.rating column, Edge Function trigger design, Athlete App dashboard display.
Ghi chú

6. Admin Web frontend: Next.js hay React SPA (Vite)?
Important▼
Next.js App Router
Public pages (scoreboard, bracket) SSR cho SEO. Admin dashboard client-side. OBS overlay là static route. Vercel deploy zero-config.

React SPA + Vite
Đơn giản hơn, không có SSR. Phù hợp nếu public scoreboard không cần SEO (chỉ share link, không cần index).
OBS overlay cần workaround cho SEO
Ảnh hưởng: Routing structure, deployment config, public embed URL pattern cho OBS overlay.

7. Referee App rewrite: giữ React (hiện tại) hay chuyển framework?
Important▼
Giữ React + add Supabase client
Codebase hiện tại (React, TypeScript) giữ nguyên. Thêm Supabase JS client. Scoring engine tái dụng hoàn toàn, chỉ thêm sync layer.
Recommended — least disruption
Rewrite thành PWA
Service worker cho offline, add-to-home-screen. Trọng tài dùng như native app. Vẫn React nhưng thêm PWA config.
+1 tuần — đáng làm vì sân hay mất mạng
Kết nối với: Referee App hiện có scoreboard full-screen view (mở tab mới) — đây chính là OBS overlay. Cần giữ feature này và expose nó qua URL pattern /overlay/{match_id} có Supabase Realtime.
Ảnh hưởng: Component sharing với Admin Web, offline queue strategy, PWA manifest.
Ghi chú
8. Athlete App rewrite: giữ React 19 + Local Storage hay migrate sang Supabase?
Important▼
Giữ codebase + swap storage layer
Tất cả UI/UX components giữ nguyên. Thay localStorage.getItem/setItem bằng Supabase queries. Recharts giữ, chỉ data source thay đổi.
Recommended — codebase hiện tại khá tốt
Rewrite → React Native / Expo
Native mobile app cho iOS/Android. VĐV có push notification thực sự. Nhưng cần build/publish lên App Store — scope lớn hơn nhiều.
v2+ — sau khi validate web version
Kết nối với: Athlete App có AI Coach dùng Gemini 2.5 phân tích 10 trận gần nhất + skills profile. Sau khi migrate sang Supabase, AI Coach có thể access tournament match results thực tế (từ Referee App) thay vì chỉ self-reported data — coaching quality tăng đáng kể.
Ảnh hưởng: Storage abstraction layer, Gemini API key management, Recharts data source.
Ghi chú
9. Gemini AI Coach: giữ client-side hay chuyển sang Supabase Edge Function?
Important▼
Supabase Edge Function
Gemini API key ở server, không expose client. Edge Function query tournament history + skills + rally stats → gửi full context cho Gemini → trả về coaching report.
Recommended — key security + richer context
Giữ client-side như hiện tại
@google/genai SDK chạy trên browser. Đơn giản, không infra thêm. Nhưng API key dễ bị lấy, và chỉ có local data.
API key risk — không nên dùng production
Kết nối với: Hiện tại Gemini chỉ thấy 10 trận self-reported + skills self-assessment. Sau khi migrate: Gemini thấy được toàn bộ tournament match log từ Referee App (rally-by-rally detail: ace, kitchen fault, unforced error...) — đây là coaching data quality ở level khác hoàn toàn.
Ảnh hưởng: Gemini API key storage, Edge Function cost (Deno runtime), prompt engineering với richer context.
Ghi chú
Feature Ownership & Data Sync 3 quyết định — ảnh hưởng DATA_SCHEMA
10. Rally detail level: Referee App sync bao nhiêu lên Supabase?
Schema▼
Full rally log sync
Sync toàn bộ: score, serve tracking, outcome (winner/error), rally reason (ace/smash/kitchen fault...), player responsible. Đây là data Referee App đang collect.
Recommended — đây là competitive moat
Chỉ sync final score + serve sequence
Admin Web chỉ cần score để advance bracket. Rally detail giữ local trong Referee App, không sync. Đơn giản hơn nhiều.
Mất đi AI coaching advantage
Kết nối với: Referee App hiện collect: ace, smash, volley, drive, drop, ATP, erne, lob, passing shot, unforced error, net error, out, service fault, kitchen fault — đây là data cực kỳ giá trị cho AI Coach. Nếu sync đầy đủ, Athlete App có player-level stats không thua gì hệ thống chuyên nghiệp.
Ảnh hưởng: rally_events table design, storage cost per match, AI Coach prompt quality.
Ghi chú
11. Skills tree: personal-only hay Admin Web cũng đọc được?
Schema▼
Personal-only — chỉ athlete thấy
Skills tree là self-assessment cá nhân, private. Admin Web không đọc được. Phù hợp với privacy expectation của VĐV.
Recommended — privacy tốt hơn
Admin đọc được skill level (có consent)
VĐV cho phép chia sẻ estimated rating với BTC. BTC thấy skill level để xếp vào bảng phù hợp (beginner/intermediate/advanced).
Useful cho skill-based seeding
Ảnh hưởng: player_skills table RLS policy, consent flow trong Athlete App, seeding algorithm trong Admin Web.

12. Shared component library: build một lib chung hay mỗi app tự manage?
Dev▼
Mỗi app tự manage — no shared lib
Copy component khi cần. Không setup monorepo, không npm link. 3 repos riêng hoặc 3 folders trong 1 repo.
Recommended — solo builder, đừng over-engineer
Monorepo + shared packages/ui
Turborepo hoặc pnpm workspace. Shared: UI components, Supabase types, scoring utils. Nhất quán hơn về design system.
+3–5 ngày setup, justify khi team lớn hơn
Kết nối với: Referee App hiện có Tailwind CSS + Recharts (giống Athlete App). Scoreboard component của Referee App có thể tái dụng trực tiếp trong Admin Web cho live display — dù không có shared lib, cần đảm bảo styling compatible.
Ảnh hưởng: Repo structure, CI/CD setup, deployment pipeline, design system consistency.
